# bodhisattva-mcp v0.1.0 — Launchable Release Design

**Date:** 2026-04-22
**Status:** Approved (ready for implementation plan)
**Target:** v0.1.0 "Launchable" — quiet Show HN / friends-and-dogfooders release

## Goal

A Python-literate person goes from zero to their first mindful pause in under 10 minutes using only the project README. The tool is installable from PyPI, CI protects the main branch, the first-run experience guides rather than frustrates, and the repo has the standard invitations for contribution (issues, PRs, CONTRIBUTING).

Framing: this tool is applied dharma — "the pause before a regrettable send." The install guide, first-run errors, and demo are all part of the invitation. Every rough edge smoothed here is an act of loving speech toward future users.

## Scope

### In scope
- PyPI package claim and publish
- Release + CI workflows on GitHub Actions
- First-run error polish at four specific failure points
- Top-level `README.md` rewrite with a staged demo GIF
- Standard repo files: `CONTRIBUTING.md`, `CHANGELOG.md`, issue templates, PR template
- Tag-based release sequence culminating in `v0.1.0`

### Out of scope
- v0.2 features (hosted framing, memory-aware pauses)
- Additional wrappers (Slack, Outlook)
- Google OAuth app verification for broad audience — v0.1 users bring their own GCP OAuth app, as documented in `docs/install.md`
- Marketing landing page, Product Hunt launch
- Refactors beyond what's needed for error polish

## Approach: Claim First, Polish On Top

Publish a pre-release (`v0.1.0a1`) as the very first step after account setup. This front-loads the riskiest unknown ("does end-to-end publishing actually work?") and lets every subsequent piece of polish be tested against the real install path.

## Components

### 1. Publishing pipeline

**Human handoff (Roger, manual, one-time):**
- Create a PyPI account at https://pypi.org
- Configure a **Pending Trusted Publisher** on PyPI (supported for projects that haven't been published yet — specify PyPI project name `bodhisattva-mcp`, GitHub owner `rogercollell`, repo `bodhisattva-mcp`, workflow filename `release.yml`). Once the first publish succeeds, the pending publisher becomes a regular trusted publisher.
- Rationale: Trusted Publishing uses short-lived OIDC tokens — no API tokens to store, rotate, or leak

**Code deliverable:** `.github/workflows/release.yml`
- Triggered on tag push matching `v*.*.*` (includes pre-releases like `v0.1.0a1`)
- Jobs:
  - `build`: checkout → setup Python 3.12 → `python -m build` → upload artifacts
  - `publish`: needs `build` → `permissions: id-token: write` → `pypa/gh-action-pypi-publish@release/v1` (OIDC)
- Version in `pyproject.toml` is the source of truth; the workflow asserts the tag matches the `project.version` and fails fast on mismatch
- No GitHub Actions environment required for v0.1 (keeps setup simple; can be added later if approval gates become useful)

**First publish:** `v0.1.0a1` — claims the name on PyPI and proves the pipeline works before any further polish lands.

### 2. CI pipeline

**Code deliverable:** `.github/workflows/ci.yml`
- Triggered on PR to `main` and push to `main`
- Matrix: Python 3.12 and 3.13 on `ubuntu-latest`
- Steps: checkout → install `uv` → `uv sync --all-groups` → `uv run ruff check .` → `uv run ruff format --check .` → `uv run pytest`
- Red run blocks merge (branch protection to be enabled manually on GitHub)

Rationale for version matrix: the project requires `>=3.12`. Testing both 3.12 and 3.13 proves forward-compat cheaply. Skipping 3.14 avoids pre-stable library churn.

### 3. First-run friendliness

Four specific failure points get wrapped with actionable error messages. Each gets a regression test so we don't silently lose the friendliness later.

| # | Failure | Where it's caught | Message points user to |
|---|---------|-------------------|------------------------|
| 1 | Missing `~/.bodhisattva/client_secret.json` | `gmail_client.py` OAuth flow init | `docs/install.md` step 2, with the expected path |
| 2 | `BODHISATTVA_LLM_PROVIDER=anthropic` but `ANTHROPIC_API_KEY` unset (and symmetric for OpenAI) | `config.py` settings validation | Exact env var name to set, and the alternative provider as fallback |
| 3 | Invalid `BODHISATTVA_LLM_PROVIDER` value | `config.py` settings validation | List of valid options (`anthropic`, `openai`) |
| 4 | Web UI port (default 8473) already in use | `web/app.py` uvicorn launch | Tells user to set `BODHISATTVA_WEB_PORT` to another port |

Each of the four gets a test in `tests/` asserting the raised error's message contains the key actionable substring (path, env var name, or config key). This prevents regression if someone later refactors the error handling.

### 4. Documentation

**`README.md` rewrite.** Top-to-bottom structure:
1. Hero: short one-liner + demo GIF
2. "What it does" — 30-second elevator pitch, preserving the v0.1/v0.2/v0.3 status line
3. Quickstart — the shortest path from install to first pause (fits on one screen)
4. "What it doesn't do" — sets honest expectations about v0.1 limits
5. Privacy one-paragraph with link to `docs/privacy.md`
6. Link to full install guide
7. License, status, contributing link

**Demo GIF.** Staged with obviously fake content (fictional characters like "Sam, about Friday's retro"). ~15–20 seconds. Shows: Claude drafts email in MCP client → Bodhisattva returns a revised draft → journal entry appears in web UI at `localhost:8473`. Stored at `docs/assets/demo.gif` and embedded at the top of the README.

If GIF production drags (capture tooling, file size, re-recording), fall back to a static 3-panel composite PNG showing the same three moments.

**`CONTRIBUTING.md`.** Covers:
- Development setup: `git clone`, `uv sync --all-groups`
- Running tests: `uv run pytest`
- Running the eval suite: `uv run python tests/evals/run_eval.py`
- Style: ruff for lint and format; commit messages follow conventional-commits (matching existing log)
- Scope note: v0.1 is intentionally small; propose v0.2+ features as issues first

**`CHANGELOG.md`.** Keep-a-Changelog format. Starts with `0.1.0a1` (pre-release), then `0.1.0`.

**`.github/ISSUE_TEMPLATE/bug_report.md`.** Fields: what happened, expected behavior, repro steps, what Claude/the agent was attempting to send (redact!), install method (`uv tool install` vs. `uv sync`), MCP client, OS, Python version, bodhisattva-mcp version.

**`.github/ISSUE_TEMPLATE/feature_request.md`.** Short: problem, proposed solution, alternatives considered, fits within v0.1 scope?

**`.github/PULL_REQUEST_TEMPLATE.md`.** Summary + test plan checkbox list.

## Release sequence

Execute in order. Each step validated before the next.

1. **Human:** Roger creates PyPI account + configures Pending Trusted Publisher
2. Land `release.yml` → bump `pyproject.toml` version to `0.1.0a1` → tag `v0.1.0a1` → verify pipeline end-to-end (see Verification below)
3. Land `ci.yml`; enable branch protection on `main` requiring CI green
4. Land first-run error polish (four failure points) + their regression tests
5. Land `README.md` rewrite + `CONTRIBUTING.md` + `CHANGELOG.md` + issue templates + PR template
6. Record + land demo GIF; embed in `README.md`
7. Bump `pyproject.toml` to `0.1.0` → tag `v0.1.0` → verify (see Verification below)
8. Update the "Status" line in `README.md` from "planned" to "shipped" for v0.1; final commit

### Verification after each publish

The MCP server runs on stdio and has no `--help` flag — it waits for MCP protocol messages. So the verification steps are:

- **After `v0.1.0a1`:**
  1. In a fresh terminal (no prior install): `uv tool install bodhisattva-mcp==0.1.0a1`
  2. `which bodhisattva-mcp` returns a path
  3. `bodhisattva-mcp --version` if we add that flag (stretch), otherwise `python -c "import bodhisattva_mcp; print(bodhisattva_mcp.__version__)"` after `uv tool install`
  4. `uv tool uninstall bodhisattva-mcp` to leave the shell clean
- **After `v0.1.0`:** same as above with `uv tool install bodhisattva-mcp` (no version pin), plus end-to-end smoke test: wire into Claude Desktop per `docs/install.md`, ask Claude to send a test email to yourself with obviously benign content, confirm `send_email` is invoked and the pause appears in the web UI journal at `localhost:8473`

## Testing

- Existing 43 tests remain green throughout
- 4 new tests (one per first-run failure point) asserting actionable error content
- CI enforces the whole suite on every PR and main push
- Manual smoke test after each publish: fresh shell, fresh virtualenv, `uv tool install` or `uvx`, launch the server, confirm the web UI is reachable at `localhost:8473`

## Rollback

- Each release-sequence step is its own PR — independently revertable
- If `v0.1.0` publish fails for any reason, `v0.1.0a1` remains usable as a fallback
- PyPI versions cannot be reused; if a bad version ships, yank it and publish `0.1.1`
- Tags can be deleted and recreated up until the publish step runs successfully

## Decisions log

- **Trusted Publishing over API tokens** — no secrets to rotate; PyPI's recommended path
- **Claim first via pre-release** — de-risks the publishing pipeline before any polish work
- **Python 3.12 + 3.13 matrix** — forward-compat proof at low cost; 3.14 skipped due to pre-stable library churn
- **Staged demo with fake content** — following Bodhisattva's own principle: real drafts might contain details you wouldn't want public later; the tool shouldn't leak what it's meant to prevent leaking
- **Each user brings their own GCP OAuth app** — the only scalable approach for v0.1 without going through Google's restricted-scope security review; documented as a feature, not a bug

## Non-goals for this spec

This spec does not cover:
- The v0.2 hosted-framing architecture
- Adding new wrap tools (Slack, Outlook, etc.)
- Changes to the framing prompt, gate logic, or journal schema
- UI/UX redesign of the web journal beyond friendlier errors

Changes in those areas are explicitly out of scope and should be separate specs.
